# Оптимизация Монте-Карло вычисления числа $\pi$

Требуется реализовать генератор `minstd_rand` и сравнить два варианта использования в задаче оценки числа $\pi$ методом Монте-Карло:

- Последовательная генерация по одному числу
- Блочная генерация через предвычисленные степени $a^k mod m$

Генератор: 
```math
x_{i+1} = a \cdot x_i \text{mod} m,
```
```math
a = 48271
```
```math
m = 2^31 - 1 = 2147483647
```


## Идея оптимизации

Для базового варианта используется обычная рекуррентная схема, где каждое следующее значение зависит от предыдущего.

В оптимизированной версии формируется блок значений:

```math
x_{i+j} = a^j * x_i mod m, \qquad j = 1..k
```

Это позволяет генерировать сразу $k$ чисел из одного базового состояния и уменьшить накладные расходы в горячем цикле Монте-Карло.

Также применена быстрая редукция по модулю Мерсенна $m = 2^31 - 1$:

```math
z = (z >> 31) + (z \& m)
```

## Решение

В файле [minstd_rand.cpp](minstd_rand.cpp) реализован бенчмарк для сравнения генераторов:
1. Целые состояния переводятся в $U(-1, 1)$
2. Проводится оценка $\pi$ по $N = 20000000$ точек
3. Оптимизированный вариант генерирует значения блоками без промежуточного буфера
4. Проводится подбор `BLOCK_SIZE` по кандидатам `8, 16, 32, 64`
5. Измеряется время в тактах CPU (`rdtsc`) и выводятся `median`/`min` циклы
6. Проверяется корректность по `hits` и разности оценок $\pi$

## Сборка и запуск

```shell
cd prac_hw/minstd_rand
make
./minstd_rand
```

Параметры сборки: `-O3 -Wall -Wextra -std=c++17 -march=native`.

## Результаты

```text
minstd_rand Monte-Carlo pi benchmark
Parameters: a=48271, m=2^31-1, seed=777, N=20000000, repeats=7
Scalar : hits=15704066, pi=3.140813, median_cycles=223413927, min_cycles=220529496
Blocked: hits=15704066, pi=3.140813, median_cycles=178769547, min_cycles=177506769, best_block_size=64
Speedup: 1.249731
Hits equal: true
Abs diff: 0.000000
```

Оба подхода дают одинаковый результат, значит оптимизированный вариант сохраняет корректность.
При подборе размера блока, лучшим вариантом оказался **64**: при сравнении по `median` получено ускорение в **1.24** раза относительно первоначального варианта. Причиной улучшения является удаление лишнего прохода по промежуточному буферу и выбор подходящего `BLOCK_SIZE` под текущий CPU.
